# Jetstream — review & discovery sweep (2026-07-09)

Multi-agent review of the current Jetstream plugin (`packages/jetstream`) + a discovery sweep
for what to build next. Method: parallel review lenses (correctness/robustness, UX/product,
code-quality/consistency) + discovery lenses (user-value, reliability/adoption), synthesized
and de-duplicated; the top findings were re-verified against source.

## Overall health

The core reducer / render / discovery design is solid. **Two structural gaps dominate the
risk** — everything else is polish:

- **A. `packages/` is outside the repo's real gate.** The root `tsconfig` / `eslint` / `vitest`
  all exclude `packages/*`, so **jetstream / manifest / profile regressions ship green through
  CI** — only a per-package `pnpm --filter` run catches them. **Fix:** run `pnpm -r check` in CI
  (or add `packages/*` to the root configs).
- **B. Discovery is CPU-only.** A process scan can classify working (high CPU) vs idle (~0 CPU)
  but **structurally cannot detect `needsInput`** (a session waiting at a prompt burns ~0 CPU),
  and it does not feed the interrupt pid map. So after a restart a genuinely-waiting session is
  painted calm blue "idle" (the doorbell never rings) and interrupt counts can disagree with
  what's signallable.

## Improvements (ranked by severity × impact ÷ effort)

Each: **[severity · effort]** — location — problem — fix.

### 1. ✅ FIXED — `restore()` clobbered live hook events after the async scan
**[HIGH · small]** — `src/state.ts` restore tail.
`restore()` replaced board state wholesale after the 4–8s `ps`+`lsof` scan; any real hook that
POSTed during the scan window was overwritten, so a session that finished mid-scan reverted to
the checkpoint's stale red "working" (with a dropped interrupt pid) and discovery — which only
fills `status==='none'` — could never downgrade it.
**Fixed:** merge under live state (`{...restored, ...this.state.sessions}` / `new Map([...pids,
...this.sessions])`) + regression test.

### 2. Page-nav strands imported boards on the bundled empty skeleton — needs a design call
**[HIGH · small]** — `src/actions/nav.ts:35`; `buildProfile` via `fixedLayout` (`src/profile.ts`).
An imported personalized board carries a "Page: Ops" nav key, but `switchToProfile` only
resolves manifest-declared bundled profiles. `Ops → Board` runs `switchToProfile('profiles/
Jetstream')` → lands on the **bundled default** board (grey "set path" keys), not the user's
imported layout — a two-tap round-trip silently drops them onto an empty skeleton.
**Fix:** omit the nav key from `buildProfile`'s imported single-page layout, OR give imports
their own PreconfiguredName-linked Ops page.

### 3. Usage key spawns `afterburner` every 60s even when no Usage key is placed
**[MED · small]** — `src/actions/usage.ts:18-19`.
`refresh()` awaits `resolveUsage()` before the `isKey()` loop; with no Usage key + a cold cache
it spawns a subprocess every `usageRefreshSec` tick for nothing (CI/heartbeat/review gate on a
placed key precisely to avoid this).
**Fix:** early-return when `![...this.actions].some(a => a.isKey())`.

### 4. "install hook" shown on a fresh install where it's already auto-wired (+ README wrong)
**[MED · small]** — `src/actions/usage.ts:38`; `README.md:20-21`.
Auto-wire installs the statusline hook on first launch; the cache only fills after `claude`
next renders, so the key immediately shows "no usage / install hook" — pointing at a
non-problem. README's "wired only by the CLI, never automatically" is now false.
**Fix:** empty-state sub → "no data yet" / "run claude"; correct the README.

### 5. In-plugin doctor re-implements PATH with a divergent dir list → false "not found"
**[MED · small]** — `src/actions/settings.ts:69` (`pluginDoctorIO`); dup in `packages/usage/src/index.ts:138-146`.
`pluginDoctorIO` inlines its own extra dirs (omits `~/.npm-global/bin`) instead of
`augmentedPath()`, so a user whose `claude`/`gh` lives there gets a permanent false "not on
PATH" warning + amber "setup N/M" face, while every real exec (which uses `augmentedPath`) finds
the binary. A third hand-rolled copy lives in the usage package.
**Fix:** build the doctor env from `augmentedPath(base.env)`; export the extra-dirs list.

### 6. Project long-press SIGINTs the session with no affordance or confirmation
**[MED · small]** — `src/actions/project.ts:60-71`.
Same key: short-press opens the editor, long-press (default 1000ms) SIGINTs the session. No
threshold affordance, no confirmation, feedback only after the SIGINT. A "jump to the project"
press held ~1s kills the current turn.
**Fix:** raise the threshold (~1500–2000ms), paint an "about to interrupt" held state past it,
and/or only allow long-press-interrupt when the project is actually `working`.

### 7. Default theme paints "working" red — indistinguishable from error/deny/stop/over-budget
**[MED · medium]** — `packages/status/src/index.ts:248` (`colorFor` `working` → `#e5484d`).
A healthy working project glows the exact red used for Deny, Stop-all-busy, failed Launch, and
usage > 90%. Under heavy multi-repo use the deck fills with red keys meaning "work in progress",
unreadable from "danger". The high-contrast theme already remaps working → orange.
**Fix:** make working blue/cyan + idle a dim neutral in the default theme (reserve red for
stop/deny/error/over-budget), or ship high-contrast semantics as default.

### 8. A session waiting since before the plugin came up shows blue "idle", not the doorbell
**[MED · medium]** — `src/state.ts:222`; `src/discover.ts:18`. (Structural gap B.)
Discovery is CPU-based and cannot detect `needsInput`; it survives restart only via `restore()`
(needs a persisted event + exactly one session per cwd). So a session that started waiting
during any update/restart is filled as blue "idle" — the doorbell never rings and won't
self-heal, misreporting the exact state the product exists to surface.
**Fix:** render discovered (hook-unconfirmed) sessions with a distinct face/glyph ("session?")
so a possibly-waiting session isn't painted confident calm blue.

### 9. `/permission` holds a queue slot + 90s timer after the client disconnects
**[MED · medium]** — `src/server.ts:70-100`; `permissions.ts:33`.
The `/permission` branch holds `res` open but only attaches disconnect handlers to `req`
(whose `close` fires right after `end`, not on the real socket drop) — there's no
`res.on('close')`. A Claude session that dies while awaiting a decision leaves a phantom pending
for up to 90s (one of `MAX_PENDING=32` slots + a live timer); pressing Approve answers a dead
socket.
**Fix:** attach `res.on('close', …)` (or thread an `AbortSignal`) to cancel the held request;
`Permissions` must expose a per-request cancel handle.

### 10. `afterburner-cli.ts` — the shared exec/PATH bedrock — has zero test coverage
**[MED · medium]** — `src/afterburner-cli.ts`.
`augmentedPath()`, `resolveAfterburner()`'s dir×ext probe, and `runAfterburner()`'s win32
`.cmd` shell-quoting are all untested; a regression silently breaks every CLI-backed key
(heartbeat/review/ci/discover/switchto) and `pnpm check` catches nothing.
**Fix:** add `afterburner-cli.test.ts` (augmentedPath output, resolveAfterburner via mocked
`existsSync`, the `.cmd` quoting branch). *(Fable-suitable.)*

### 11. `ProjectKey` carries real untested face/state logic + a loosely-typed status map
**[MED · medium]** — `src/actions/project.ts:28,116-159`.
`subLine()`, `trackDiff()`'s done-episode cache/invalidation, and `glyph()` are nontrivial with
no `project.test.ts`; `STATUS_LABEL` is `Record<string,string>` not `Record<ProjectStatus,
string>`, so a renamed status silently falls through to `''`.
**Fix:** extract the decision logic into a pure tested module (à la `encoder.ts`); tighten
`STATUS_LABEL` to `Record<ProjectStatus, string>`.

### 12. InterruptAll / Project long-press report a "working" count they can't signal
**[LOW · medium]** — `src/actions/interrupt-all.ts:19,24`; `src/state.ts:164` (`allPids`).
`renderAll` counts `byProject()` "working" (which includes discovery-only sessions) but
`onKeyDown` signals `interruptPids(board.allPids())`, and `allPids` returns only hook-noted pids
— never discovered ones. A CPU-visible session with no noted pid paints red "N working" yet
interrupt returns 0 → showAlert.
**Fix:** base the working-count on sessions with a signallable pid (or track discovered pids).

### 13. Mechanical duplication cluster (the package's main maintainability tax)
**[LOW · medium]** — various.
`errMsg` inlined ×10; placed-key check 3 ways (`ci.ts:57` / `heartbeat.ts:35` / `review.ts:30`);
PATH-probe loop ×3 with divergent win-ext semantics (`afterburner-cli.ts:25` / `switchto.ts:28`
/ `doctor.ts:144`); near-identical `DiffStatExec`/`CiExec` seams; uniform-face paint loop copied
~14×; dead `UsageKey.feed` field; stale `formatCountdown` doc; duplicated readline wiring
(`cli.ts:124` vs `:148`).
**Fix:** extract shared helpers (`errMsg`, `hasPlacedKey(actions)`, one `resolveOnPath`, a shared
`Exec` factory, `paintKeys(actions, face)`, an `AfterburnerNotInstalledError` sentinel); drop
the dead field + stale doc. *(Fable-suitable.)*

### 14. Doc drift: README "jumps to the terminal"; generated profile written into the plugin bundle
**[LOW · small]** — `README.md:44` vs `switchto.ts:53`; `init.ts:203` / `cli.ts:169` vs `settings.ts:225`.
`buildOpenCommand` opens the project **folder** in an editor and never launches a terminal.
Separately, init/chat write `Jetstream.streamDeckProfile` to `process.cwd()`; the README tells
users to run init from inside the plugin folder, so the importable artifact lands in the
`.sdPlugin` bundle (wiped on every plugin update), while the in-app path correctly uses
`~/Downloads`.
**Fix:** reword to "opens the project in your editor"; write the generated profile to
`~/Downloads` and print that path.

## Build-next (discovery, ranked by impact ÷ effort)

1. **Always-Allow key** — **[HIGH · medium]**. A third permission key that settles the head
   request `allow` AND records a **session-scoped** rule (tool + session, optionally the command
   prefix from `summarizeTool`); `request()` auto-settles later matching requests with no
   keypress; rules expire on `SessionEnd` (reuse the sessions map); active count on the Settings
   face; a toggle clears all. **Why now:** `permissions.ts` holds every request open for a
   physical press with zero allow-memory — a many-repo driver re-taps the same safe Read/Grep/
   Bash-test prompts all day. The scaffolding (`summarizeTool`, SessionEnd tracking) already
   exists. The single biggest repeated-friction kill.

2. **Dark-board self-diagnosis** — **[HIGH · medium]**. Add a listener-liveness probe to
   `doctor.ts`: `GET 127.0.0.1:<port>/health` (~500ms) reporting the hook listener up vs "not
   running / port in use"; wire into `runDoctor()` + `pluginHealth()`. On the failure side,
   replace `plugin.ts`'s swallow-to-logger bind handler with a retry/fallback + a visible "hooks
   offline" face. **Why now:** the `/health` endpoint already exists (`server.ts:51`) but is used
   nowhere; a bind failure is only logged to a file no user opens. The #1 "board stays dark"
   cause (port 41321 already held) is completely invisible — doctor reports all-green while zero
   events flow.

3. **Focus the live terminal on project press** — **[HIGH · medium]**. When a Project key (or
   Fleet-dial tap / Attention press) is pressed and `board.pidsForProject(id)` has a live
   session, raise **that** session's terminal window (osascript: pid → tty → owning Terminal/
   iTerm/Ghostty window); fall back to the current editor-folder open otherwise. **Why now:** the
   core promise is press-to-jump-into-that-project's-Claude — an amber needs-you key should
   surface the terminal where Claude is asking, not spawn a new editor window. pid+cwd are
   already stored per session; the plugin is mac-only so AppleScript is always available.

4. **Doorbell ack + per-project snooze + quiet hours** — **[HIGH · medium]**. PagerDuty
   semantics for the Attention key: config `quietHours`, a board-side per-project snooze map;
   tap = ack (stop flash, keep amber), long-press = snooze the neediest project 1h;
   `shouldEscalate()` gates on quiet hours + snooze. **Why now:** `attention.ts` only stops
   flashing when the project stops needing input — one genuinely-blocked repo flashes forever,
   at 3am. `config.ts` has `escalateAfterSec` but no quiet hours / snooze. (Roadmap #5 + #13.)

5. **Stale / stuck-session status** — **[MED · small]**. When a session has been `working` past a
   threshold (~20m) render a warning glyph / dim tint; sharpen with discovery — a hook-`working`
   project whose discovered CPU has fallen to idle is likely stalled. **Why now:** the board
   renders a 5-minute and a 5-hour `working` identically, so a hung session is invisible until
   you notice it never turned green. `board` tracks `since` and `discover.ts` already computes
   active-vs-idle CPU — the signal exists and is unused.
